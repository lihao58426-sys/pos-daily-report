# POS 日报推送系统 — 技术文档

> 版本：V1.0
> 日期：2026年7月10日
> 用途：银豹后台数据自动抓取 + SQLite 持久化 + 企业微信日报推送

---

## 一、项目概述

自动登录银豹 POS 后台 → 抓取营业实收、商品排名、储值卡/次卡/会员付费升级数据 → 存入 SQLite → 拼 Markdown 日报 → 推送到企业微信群。支持多门店、反检测、定时调度、演习模式。

---

## 二、技术栈

| 层面 | 技术 | 说明 |
|------|------|------|
| 浏览器自动化 | Playwright | Chromium 模拟登录、页面解析 |
| 数据库 | SQLite（sqlite3） | Python 原生支持，零配置 |
| 推送 | 企业微信 Webhook | Markdown 格式消息 |
| 配置 | YAML | 多门店列表、运行参数 |
| 日志 | logging | 时间+级别+模块名，文件+控制台双输出 |
| 定时任务 | 自调度算法 | 每天随机时间执行，41天池不重复+秒级去重 |
| 反检测 | 浏览器指纹隐藏 + UA伪装 + 真人鼠标轨迹 |

---

## 三、系统架构

```
main.py（厨师长，只管调度）
  │
  ├─ config.py     → 读 YAML + 环境变量
  ├─ crawler.py    → Playwright 登录 → 抓三处数据
  ├─ models.py     → dict → DailyReport dataclass
  ├─ database.py   → 存入 SQLite（双表）
  ├─ report.py     → 拼 Markdown 日报
  └─ pusher.py     → HTTP POST 到企微群

数据流：
  银豹后台 → crawler → dict → DailyReport → database → report → pusher → 企微群
                                    ↓
                            SQLite（daily_reports + product_rankings）
```

## 五、核心功能

### 5.1 数据抓取

| 抓取项 | 来源 | 方法 |
|--------|------|------|
| 营业实收 | 首页 Dashboard | `_extract_value_after_label` 文本解析 |
| 商品排名 Top5 | 首页滚动→商品消费单数排名 | `_parse_product_ranking_from_page` Playwright选择器 |
| 储值卡/次卡/会员升级 | 营业概况明细页 | 直接导航→ `span[data-show-name="现金支付"]` |

### 5.2 多店支持

config.yaml 配置门店列表 → main.py 循环遍历 → crawler 每店独立 Cookie → database 每行数据打 store_id 标签 → 单店零配置向后兼容

### 5.3 反检测

| 措施 | 实现 |
|------|------|
| 浏览器指纹隐藏 | `--disable-blink-features=AutomationControlled` |
| UA 伪装 | Chrome 131 / Windows 10 |
| 真人鼠标 | `_human_move()` 分 10-25 步 + 随机抖动 |
| 随机延迟 | 每次操作后 1-3 秒随机停顿 |

### 5.4 定时调度

自调度算法：每次执行后计算下次随机时间 → 41 分钟数池用完才重复 → 秒级随机 → 银豹日志看不出规律。代码已写注释，待上云激活。

## 七、开发踩坑记录

| 问题 | 原因 | 解决 |
|------|------|------|
| "更多"按钮点不动 | 在 dashboard 上找"更多"，实际在新标签页打开 | 直接导航到 `/Report/BusinessSummaryV2` |
| 商品排名解析空 | 正则匹配 `"xxx 120单"` 格式，实际 HTML 是结构化 div | 改用 Playwright 选择器定位 `.percentBar__top-left` |
| 会员付费升级提取失败 | 该行 `<tr>` 没有 `data-row-key` 属性 | 改用 `tr:has(a:has-text("会员付费升级"))` |
| `page.inner_text()` 报错 | 无参数调用需要 selector | 改为 `page.inner_text("body")` |

## 九、测试策略

### 测试覆盖

```
test_models.py      6 passed    数据解析（正常/空/缺字段/坏数字/含排名/收入明细）
test_report.py      5 passed    日报构建（基本/含收入/含时间/自定义日期/零营收）
test_database.py    5 passed    数据存储（插入/趋势/汇总/多店隔离/商品排名入库）
test_pusher.py      5 passed    推送逻辑（成功/失败/网络异常/超时/请求格式验证）
─────────────────────────
                  21 passed    0.06s 全部通过
```

### 测试原则

- 从底层往上测：models → report → database → pusher
- 数据层用 `:memory:` 临时数据库，不碰真实数据
- 网络层用 `unittest.mock` 模拟 HTTP 请求，不真发企微消息
- `python -m pytest -v` 一键跑全部

### Mock vs Dry-run

| | Dry-run | Mock 测试 |
|------|---------|---------|
| 场景 | `python main.py --dry-run` 手动跑 | `pytest` 自动跑 |
| 联网 | 要（Playwright 真的登银豹） | 不要（requests 被替换成假的） |
| 能测什么 | "整个流程通不通" | "每个函数逻辑对不对" |
| 能测失败场景吗 | ❌ 没法让企微真的返回错误 | ✅ mock 返回什么由你控制 |
